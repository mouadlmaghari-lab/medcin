## 1. Executive Summary

TabibCare is a cloud-based medical practice management SaaS platform designed for solo medical practitioners in Morocco. The platform transitions an existing, validated desktop application into a modern web and mobile solution, serving doctors and their patients through distinct role-based interfaces:

- A **Doctor web interface** with full access to all medical and administrative modules
- A **Medical Assistant (Aide-Medecin) web interface** with access to scheduling, patients, payments, and expenses
- A **Patient mobile application (Espace Patient)** for appointment booking, document access, and multi-doctor management

The platform launches with a focus on diabetology/endocrinology (leveraging existing consultation fields for HbA1c, glycemia, lipid panels, etc.), with architecture designed to be specialty-agnostic for future expansion. TabibCare addresses a significant gap in the Moroccan healthcare technology market where the majority of solo practitioners still rely on Excel or paper.

**Key Differentiator:** The patient mobile app creates a network effect. Patients using the app with one doctor can connect with other doctors on the platform, driving organic growth.

## 3. Target Market & Users

### 3.1 Primary Target

Solo medical practitioners in Morocco, starting with diabetologists/endocrinologists. The majority of this market currently uses Excel or paper-based systems, representing a large untapped opportunity.

### 3.2 User Personas

#### Persona 1: Dr. Karim (Diabetologist / Doctor Role)
- Age 35-55, solo practice in urban Morocco
- Sees 15-25 patients daily
- Needs to track HbA1c, glycemia, lipid panels, weight evolution over time
- Currently uses a mix of Excel and paper
- Wants digital ordonnances and certificates with his cabinet branding
- Pain points: time wasted on admin, lost patient files, no analytics

#### Persona 2: Fatima (Medical Assistant / Aide-Medecin Role)
- Manages the front desk: scheduling, patient intake, payments
- Uses the simplified assistant interface (teal/blue navigation bar)
- Access limited to: **Rendez-vous, Patients, Reglements, Depenses**
- Does NOT see consultations, ordonnances, certificates, reports, or expertises
- Processes payments and prints invoices/receipts

#### Persona 3: Ahmed (Patient / Mobile App User)
- Visits multiple doctors (diabetologist, cardiologist, GP)
- Wants to book appointments without calling the office
- Needs to access prescriptions and certificates on his phone
- Wants to upload lab analyses for doctor review before visits
- Values having all medical records from different doctors in one app

## 7. Patient Mobile App (Espace Patient)

The patient mobile app is both a feature and a growth engine. It creates network effects by allowing patients to manage their relationship with multiple doctors on the platform.

### 7.1 Core Features

- Self-registration and profile management
- Book and cancel appointments with any connected doctor
- View upcoming and past appointments across all connected doctors
- Access prescriptions/ordonnances (read-only, PDF download)
- View medical certificates (read-only, PDF download)
- View consultation history per doctor (with doctor permission)
- Upload medical analyses and lab results for doctor review
- Push notifications via OneSignal
- Multi-doctor dashboard: see all connected doctors in one view

### 7.2 Patient-Doctor Linking

#### Scenario A: Patient already has the app
1. Patient arrives at doctor's office
2. Secretary searches by phone or CIN
3. System finds existing patient account
4. Doctor sends link request
5. Patient approves in their app
6. Records now visible to this doctor (per patient permissions)

#### Scenario B: Patient does not have the app
1. Secretary creates a physical patient record (name, phone, CIN)
2. System generates an invitation link/SMS
3. Patient downloads app and registers
4. Patient claims existing record by verifying phone/CIN

#### Scenario C: Physical-only patient (no app needed)
1. Secretary creates patient record in doctor system
2. Patient exists only in the doctor's database
3. No mobile app interaction required
4. Patient can claim account at any time in the future

### 7.3 Patient Permission System

Patients control what each connected doctor can see:

- **Default:** each doctor sees ONLY their own records
- Patient can opt-in to share consultations cross-doctor
- Lab result uploads: patient chooses which doctors can view
- Prescriptions: visible to prescribing doctor; patient can share with others

### 7.4 Analyses Upload

- Supported formats: PDF, JPEG, PNG
- Max file size: 10 MB per file
- Stored on Backblaze B2 with signed URLs
- Patient selects which doctor(s) to share with
- Doctor receives push notification on new upload

## 8. File Storage & Document Management

### 8.1 Storage Architecture

| Field | Value |
|---|---|
| **Provider** | Backblaze B2 (S3-compatible API) |
| **Access Method** | Pre-signed URLs (time-limited, secure) |
| **Bucket Structure** | One bucket per environment (prod/staging) |
| **File Organization** | `/{tenant_id}/{patient_id}/{category}/{filename}` |
| **Encryption** | AES-256 at rest + TLS in transit |
| **Max File Size** | 10 MB (patient), 25 MB (doctor) |
| **Retention** | Indefinite (medical record requirements) |

### 8.2 File Categories

| Category | Generated By | Access |
|---|---|---|
| Patient Photos | Doctor/Secretary | Doctor, Patient |
| Ordonnances (PDF) | System (auto-gen) | Doctor, Patient (read-only) |
| Certificates (PDF) | System (auto-gen) | Doctor, Patient (read-only) |
| Invoices (PDF) | System (auto-gen) | Doctor, Patient |
| Medical Reports | Doctor | Doctor, Patient (if shared) |
| Lab Analyses | Patient (upload) | Patient, Authorized doctors |
| Expertise Attachments | Doctor | Doctor only |
| Expense Receipts | Doctor/Secretary | Doctor only |

### 8.3 Security

- Pre-signed URLs with 15-minute expiration
- No public bucket access; all requests go through Laravel API
- MIME type + extension validation on upload
- Audit log for all file access

## 9. Multi-tenancy Architecture

### 9.1 Single Database, Tenant-per-Row

All doctor/clinic data stored in one database with `tenant_id` on every tenant-scoped table. Optimal for hundreds to low thousands of solo practitioners.

**Tenant-Scoped (isolated per doctor):**
- Patients, Appointments, Consultations, Certificates, Prescriptions
- Invoices, Payments, Expenses, Medications/Stock
- Medical reports, Expertises, Evolution notes

**Global (shared across tenants):**
- Patient accounts (one patient, many doctors)
- Medication reference catalog
- Cities, system configuration

### 9.2 Data Isolation

- Laravel Global Scopes auto-apply `tenant_id` filter
- Middleware sets tenant from authenticated user context
- Patient app uses separate permission layer (not tenant-scoped)

## 12. Security & Compliance

### 12.1 Authentication

- Laravel Sanctum for API tokens + tymon/jwt-auth for mobile
- bcrypt password hashing (cost factor 12)
- Optional 2FA (TOTP) for doctor accounts
- Session: 8h web, 30 days mobile (refresh token)

### 12.2 Data Protection

- HTTPS (TLS 1.3) for all API communication
- Database + file encryption at rest
- Sensitive fields encrypted at app level (CIN, medical data)
- Pre-signed URLs with short expiration

### 12.3 Moroccan Compliance

- **Law 09-08:** consent, data minimization, right to access/rectify
- **Law 05-20:** security measures, incident response
- Patient consent recorded digitally when linking to a doctor
- Data export/deletion for right-to-be-forgotten

### 12.4 Audit & Logging (Spatie Activitylog)

- All user actions logged (who, what, when, IP)
- Medical record access logged (read and write)
- Modification history for patient records
- Log retention: 5 years

## 16. API Design

- **Doctor API** (`/api/v1/doctor/*`): Full-access endpoints, tenant-scoped
- **Assistant API** (`/api/v1/assistant/*`): Limited endpoints (appointments, patients, payments, expenses)
- **Patient API** (`/api/v1/patient/*`): Mobile app endpoints, permission-scoped

GET/POST   /api/v1/doctor/certificates
